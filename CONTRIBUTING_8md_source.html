4 When contributing to this repository, please first discuss the change you wish
5 to make via issue, email, or any other method with the owners of this repository

9 ## Pull Request Process
10 
11 1. Create a pull request early in development processes with a checkbox list of
12  features/changes you plan to make along with a description of what you intend
13  to do. Prepend (WIP:) for "work in progress" to the title of this pull request
14  until you are ready for review and merging.
15 2. Update README.rst and the Sphinx documentation with detains of any changes to
16  the interface, this includes new environment variables, exposed ports, and
17  useful file locations.
18 3. Ensure test coverage did not drop below 90% and that all unit tests pass.
19 4. Ensure the package builds, installs, and runs using `pip install .` in
20  a clean `virtualenv`. List any new external dependencies in the README.rst
21  file.
22 5. Increase the version numbers in any examples files and the README.rst to the
23  new version that this Pull Request would represent. The versioning scheme we
24  use is [SemVer](http://semver.org/).
25 6. You may merge the Pull Request in once you have the sign-off of a core
26  developer, or if you do not have permission to do that, you may request the
27  reviewer merge it for you.

30 ## Git Etiquette
31 
32 * Do not work on the main repository unless you are merging in changes,
33  preparing for a version bump, or other repository management.
34  * Create a fork, make a new branch and work on your changes there. Merge
35  these changes in using the _Pull Request Process_ above.
36 * Do not commit IDE/Editor specific files or any generated files. You should
37  add these files to the `.gitignore` file to ensure you do not accidentally
38  commit them.
39 * Do not commit application specific configuration files. If you wish to submit
40  an example place it in the examples directory.
41 * Do not commit assets such as pictures or videos. The exception is small
42  (under 300 kB) images used in the documentation.
43 * Do not commit other project's documentation such as PDF files.
44  * Link to that projects documentation instead.
45 * Do not commit another project's source code.
46  * Use a [Git submodule](https://git-scm.com/docs/git-submodule) instead.

49 ## Programming Style
50 
51 * All C++17 features are allowed.
52 * Simply running the `style` rule of the Makefile will fix most style issues and
53  should be used to ensure consistent style. This target will be ran before
54  a merge to check for style consistency but does not check naming conventions.
55 * All type names such as classes, structs, type aliases, enums, and type
56  template parameters should be `CamelCase`.
57 * Functions, variables, and namespaces should be `snake_case`.
58 * No name may begin with a single `_` (underscore) or contain `__` (double
59  underscore).
60 * Private data members and functions should end in an underscore.
61  have an `autodoc` entry in the Sphinx documentation.
62 * File name must be lower case and should only include letters, numbers, and
63  underscores.
64 * C++ source code files should use the `.cpp` extension and `.hpp` for header
65  files. `.c` and `.h` should only be used for C files.
66 * All header files must have include guards and the name should be the file name
67  (all uppercase) with `_HPP_` appended.
68 * All public functions and classes should contain [Doxygen style
69  comments](https://www.stack.nl/~dimitri/doxygen/manual/docblocks.html) and the
70  recommended block format is
71 
72  \**
73  *
74  */
75 
76  And the `\` (backslash) is recommended over the `@` (at symbol).
